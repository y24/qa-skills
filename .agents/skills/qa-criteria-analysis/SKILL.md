# 品質基準策定(非機能クライテリア分析)

QAの視点で、非機能に関する**品質基準項目・判定基準・確認方法**を策定するスキル。
工程移行判定やリリース判定の品質評価で「満たしていることを確認すべき項目」を
迅速に揃えることが目的。

テスト分析(フェーズ2)が「何をどの深さでテストするか」という方針で終わるのに対し、
このスキルは**合否を判定できる基準**まで落とす。成果物はテスト計画の完了基準
(品質ゲート)とテスト観点抽出の導出元になり、単独ではリリース判定会議の
判定材料としてそのまま使える。

## 実行前に読むこと

- [_shared/conventions.md](../_shared/conventions.md)
- [_shared/references/quality-characteristics.md](../_shared/references/quality-characteristics.md)(分類の骨格。全基準をこの特性で分類する)
- 前フェーズの成果物: テスト分析(選定済み品質特性・リスク評価)、不具合分析(あるものすべて)

## 入力

- 開発ドキュメント(いずれか複数): プロジェクト計画書 / 要件定義書 / 仕様書 /
  PR差分 / その他ドメイン知識
- (あれば)テスト分析レポート、不具合分析レポート、既存の品質基準・判定実績

## 手順

1. **インプット資料の確認**(複数選択): プロジェクト計画書 / 要件定義書 / 仕様書 /
   PR差分 / 過去不具合一覧 / 既存の品質基準 / その他。選択後、各資料のパスを確認する。
2. **品質リスク抽出**: 資料を読み、quality-characteristics.md の全特性(ISO 25010
   主特性+業務システム拡張観点)を上から走査して「関係する / 関係しない / 不明」に
   分類する。テスト分析レポートがあればその選定結果を引き継ぎ、差分(資料から新たに
   見えたリスク)だけを追加確認する。各リスクに evidence_level と根拠(資料の節・
   PR の該当箇所)を付ける。
3. **品質基準項目の策定**: 「関係する」特性ごとに基準を定義する。1基準につき:
   - **品質基準項目**: 何を満たすべきか(例: 月次締め処理のバッチ処理時間)
   - **判定基準**: 合否を第三者が機械的に判定できる条件。定量が原則
     (例: 伝票10万件で30分以内)。定性基準の場合は判定手続き(誰が何を見て合と
     するか)まで書く。
   - **確認方法**: 何を・どの環境で・どう測定/確認するか
   - **確認工程**: 設計 / UT / IT / ST(複数可)。その基準を**最も早く・安く
     確認できる工程**に割り当てる(シフトレフト)。設計レビューで確認できる
     ものを ST に回さない。
   - **優先度**: must(未達なら工程移行・リリース不可)/ should(未達は判定の
     場で協議)
4. **策定例の生成(限定許可)**: 資料から判定基準の具体値を決められない場合に限り、
   一般的な水準や類似システムの慣行に基づく**策定例**を生成してよい。ただし:
   - `basis: example` を必ず付け、requirement / derived と混ぜて並べない。
   - 成果物の冒頭に「策定例を N 件含む」旨の注記と、策定例だけの再掲一覧を置く。
   - 例の根拠(何に基づく水準か)を1行で書く。根拠を書けない値は出さない。
5. **基準の出所(basis)の付与**: 全基準に以下のいずれかを付ける。
   | basis | 意味 | evidence_level |
   |---|---|---|
   | requirement | 要件・計画書に明記された値 | confirmed |
   | derived | 資料・過去実績から導出した値 | likely |
   | example | 情報不足のため生成した策定例 | hypothesis |
6. **ユーザー確認**(選択式): 次を確認してから成果物を確定する。
   - 「関係しない」と除外した特性の承認
   - 策定例それぞれの扱い: 「この値で要件として確定 / 例のまま保留(関係者に確認) /
     削除」
   - must / should の割り当て
7. **承認ゲート**(conventions.md §4)。

## 出力フォーマット(NN-criteria-analysis.md)

```markdown
# 品質基準一覧: <対象>

> ⚠️ この文書には情報不足のため生成した**策定例が N 件**含まれる(basis: example)。
> 策定例は関係者の合意を得るまで判定に使用しないこと。
(策定例が0件ならこの注記を削除)

## 1. 対象と範囲
(対象機能・変更の範囲)

## 2. 品質リスクサマリ
| 品質特性 | 関係 | リスク概要 | 根拠 | evidence_level |

## 3. 品質基準一覧(品質特性別)

### <品質特性名>
| ID | 品質基準項目 | 判定基準 | 確認方法 | 確認工程 | 優先度 | basis | 根拠 |

- ID は `QC-<特性略号>-NN`(例: QC-PERF-01, QC-AUDIT-02)
- 確認工程: 設計 / UT / IT / ST(複数可)

## 4. 策定例一覧(basis: example の再掲)
| ID | 判定基準(例) | 例の根拠 | ユーザー判定(確定/保留/削除) |

## 5. 除外した品質特性と理由

## 6. 不足情報
(基準化できなかったリスク、確認が必要な事項、誰に確認すべきか)
```

## 品質基準

- 判定基準は「速い」「十分な」等の曖昧語で書かない。合否が割れる表現を見つけたら
  qa-spec-review モード2の検出フォーマットで記録し、セクション7に残す。
- basis: example を requirement / derived と区別なく並べない。冒頭注記・再掲一覧の
  ない成果物に策定例を含めてはならない。
- 「全特性に基準を作る」ことを網羅とみなさない。関係しない特性は理由付きで除外し、
  レビュー時の反証対象として残す。
- 確認工程のラベルがない基準を出さない。「どの工程でも確認できる」は
  最も早い工程に倒す。
- インプット資料にない値を basis 明示なしに補完しない(conventions.md §7)。
