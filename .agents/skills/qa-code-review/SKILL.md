# QAコードレビュー(品質特性ベースのシフトレフト)

「仕様通りに動くか」「読みやすいか」を見る開発者レビューとは別物。**品質特性の
観点でコードを走査し、テストという手段に頼る前に品質リスクを潰す・テストの濃淡を
決める材料を作る**のが目的。

qa-feature-investigation(フェーズ6)がコードから「仕様の材料」を集めるのに対し、
このスキルはコードから「品質リスク」と「保証済み事項」を検出する。同じコードを
読むが目的が違うため、機能調査レポートがあれば処理の流れの再調査は省略できる。

## 実行前に読むこと

- [_shared/conventions.md](../_shared/conventions.md)
- [_shared/references/quality-characteristics.md](../_shared/references/quality-characteristics.md)(特性の一覧と選定手順)
- [_shared/references/code-review-viewpoints.md](../_shared/references/code-review-viewpoints.md)(特性ごとのコードレベル・チェックポイント)
- [_shared/references/regression-viewpoint-catalog.md](../_shared/references/regression-viewpoint-catalog.md)(過去不具合パターン)
- [_shared/knowledge-source.md](../_shared/knowledge-source.md)(対象機能の業務背景を wiki で把握してから読む)
- 前フェーズの成果物: テスト分析(選定済み品質特性・リスク評価)、機能調査レポート(あるものすべて)

## 入力

- 対象コード: PR・diff(推奨。変更起点でスコープを絞れる)またはディレクトリ・機能単位
- (あれば)仕様書、DBスキーマ、機能調査レポート

## 手順

1. **スコープ確定**(選択式): レビュー範囲を確認する。
   - PR・diff のみ(推奨: 変更行+その呼び出し元・呼び出し先まで)
   - 機能全体
   - 特定の品質特性に絞った横断走査(例: 計算正確性だけを機能横断で)
2. **品質特性の選定**: テスト分析(フェーズ2)があればその選定結果を引き継ぐ。
   なければ quality-characteristics.md の使い方に従い「関係する / 関係しない / 不明」に
   分類し、選択式で承認を得る。**走査しなかった特性と理由は成果物に残す**。
3. **特性別走査**: 選定した特性ごとに code-review-viewpoints.md のチェックポイントを
   コードに当てる。指摘には必ず出典(ファイルパス:行)を付ける。コードで確認できた
   ものだけが confirmed、呼び出し先未読などで確証がないものは likely / hypothesis。
4. **回帰観点カタログとの突合**: カタログの各エントリの「発生しやすい条件」が
   対象コードに潜んでいないか確認する(該当した指摘には根拠チケットIDを引き継ぐ)。
5. **シフトレフト判定**: 検出事項と主要な確認ポイントを次の3つに仕分ける。ここが
   このスキルの中核。
   - **実装修正を推奨**: リスクが具体的で、テストで検出するより直す方が安い
   - **テストで確認が必要**: コードだけでは白黒つかない(→ テスト観点へ引き継ぐ)
   - **コードで保証済み**: 型・制約・共通処理等で担保されており、テストを軽減できる
     (根拠となるコード箇所を必ず示す)
6. **指摘の整理**: 各指摘に深刻度(Critical/Major/Minor)、品質特性ラベル、
   シフトレフト判定、修正提案、evidence_level を付ける。良い実装(品質を担保している
   共通化・ガード)も挙げる — 「コードで保証済み」の根拠になる。
7. **フィードバック**(選択式): 指摘ごとに「開発へ連携する / テスト観点に回す /
   対応不要(理由あり)」をユーザーに確認する。**コードの修正はこのスキルでは行わない**。

## 出力フォーマット(NN-code-review.md)

```markdown
# QAコードレビュー: <対象>

## 1. レビュー範囲と走査した品質特性
(走査しなかった特性とその理由を含む)

## 2. 指摘一覧
| # | 深刻度 | 品質特性 | 指摘 | 出典 | シフトレフト判定 | 修正提案/推奨テスト観点 | evidence_level |

## 3. コードで保証済みの事項(テスト軽減候補)
| 事項 | 保証の仕組み | 出典 | 軽減できるテスト |

## 4. テスト観点への引き継ぎ(→ qa-test-viewpoint)
(指摘由来の推奨テスト観点。品質特性ラベルと出典を保持したまま渡す)

## 5. 回帰観点カタログとの突合結果

## 6. ユーザー判定と対応結果

## 7. チェックポイント更新の提案
(今回の指摘のうち code-review-viewpoints.md に追記すべき頻出パターン)
```

## 品質基準

- 出典(ファイルパス:行)のない指摘を書かない。
- 命名・スタイル・可読性など開発者レビューの領域に脱線しない。保守性の指摘は
  「テスト容易性・解析性を損なう構造」に限定する。
- 「コードで保証済み」は指摘以上に慎重に。根拠コードを示せないものをテスト軽減候補に
  入れない(誤った軽減は指摘漏れより有害)。
- 指摘ゼロの場合は「どの特性を・どのチェックポイントで・どこまで確認したか」を明記する。
- バグの疑いを見つけても修正実装に脱線しない。指摘として記録し、判断は人間に委ねる。
