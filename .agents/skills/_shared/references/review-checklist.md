# テスト設計レビューチェックリスト

qa-test-design-review が使うレビュー観点。**「自分がレビューでよく指摘すること」を
蓄積する場所**であり、一般論のチェックリストで終わらせないこと。
qa-improvement 実行時に、そのセッションで出た指摘をここへ還元する。

## A. 網羅性

- [ ] 仕様書の全機能・全条件がテスト観点に対応づいているか(トレーサビリティ)
- [ ] 無効系・異常系が有効系と同等の密度で設計されているか
- [ ] 境界値が「境界・境界±1」まで展開されているか
- [ ] 0件・1件・大量件のデータ量バリエーションがあるか
- [ ] 権限なしユーザー・他者データへのアクセスが試されているか

## B. 過去不具合の反映

- [ ] [regression-viewpoint-catalog.md](regression-viewpoint-catalog.md) の該当観点が反映されているか
- [ ] 対象機能・類似機能の過去不具合チケットが確認されているか
- [ ] 今回の変更が過去に回帰を起こした箇所に触れていないか

## C. 期待結果の質

- [ ] すべての期待結果が合否判定可能か(「正しく〜」「適切に〜」が残っていないか)
- [ ] 期待結果の根拠(仕様書の節番号等)が辿れるか
- [ ] 画面・帳票・エクスポートの数値一致など、複数出力間の整合が確認されるか

## D. 過剰・重複

- [ ] 同じ同値クラスを叩くだけの重複ケースがないか
- [ ] リスクの低い箇所に過剰な組み合わせテストをしていないか
- [ ] 自動化済みの回帰と手動ケースが無駄に重複していないか

## E. 実行可能性

- [ ] 必要なテストデータの作成方法が明確か
- [ ] 環境・権限・外部連携の前提条件が明記されているか
- [ ] 実行順序の依存(データを壊すケースが先行するなど)が整理されているか

## F. 自分の頻出指摘(ここを育てる)

<!-- レビューで繰り返し指摘したことをここに追記する。例: -->
- [ ] (例)再実行・二重送信の観点が抜けていないか
- [ ] (例)Excel出力の書式・ロケール観点が抜けていないか

## 評価基準(S/A/B/C/D)

| 評価 | 基準 |
|---|---|
| S | 指摘なし。過去不具合観点まで反映済み |
| A | 軽微な指摘のみ。そのまま実行に進める |
| B | 中程度の抜けあり。修正後に再レビュー不要 |
| C | 重要な抜けあり。修正後に再レビュー必要 |
| D | 設計の前提(仕様理解・スコープ)から誤り。やり直し |
