# テスト設計レビュー(独立レビュー)

作成者とは別の視点でテスト設計を検証するスキル。**このフローの前フェーズで自分
(AI)が作った成果物であっても、擁護せずゼロベースで粗を探す**。

## 実行前に読むこと

- [_shared/conventions.md](../_shared/conventions.md)
- [_shared/references/review-checklist.md](../_shared/references/review-checklist.md)(観点A〜FとS〜D基準)
- [_shared/references/regression-viewpoint-catalog.md](../_shared/references/regression-viewpoint-catalog.md)
- 仕様書・テスト分析・仕様レビュー結果(入手できるものすべて)

## 入力

- レビュー対象: テスト観点一覧 / テストケース(どちらか、または両方)
- 人間が作った設計書でもよい(フォーマットは問わない)

## 手順

1. **仕様側からの突合(フォワード)**: 仕様書の機能・条件を列挙し、それぞれに対応する
   観点・ケースが存在するか確認する。対応のない仕様記述が「抜け」候補。
2. **設計側からの突合(バックワード)**: 各観点・ケースの導出元が辿れるか、期待結果が
   仕様と一致するか確認する。導出元不明のケースが「仕様理解誤り」候補。
3. **チェックリスト走査**: review-checklist.md のA〜Fを順に適用する。
4. **過去不具合の反映確認**: 回帰観点カタログの該当エントリが反映されているか確認する。
5. **指摘の整理**: 各指摘に深刻度(Critical/Major/Minor)、根拠、修正案、
   evidence_level を付ける。良い点も3つ以上挙げる(何を維持すべきかの情報になる)。
6. **総合評価**: review-checklist.md の基準で S/A/B/C/D を付け、評価理由を書く。
7. **フィードバック**: 指摘ごとに「修正する / 修正しない(理由あり) / 保留」を
   選択式でユーザーに確認し、「修正する」が選ばれたものは修正を実施する。

## 出力フォーマット(NN-test-design-review.md)

```markdown
# テスト設計レビュー: <対象>

## 1. 総合評価: S/A/B/C/D
(評価理由 3〜5行)

## 2. 指摘一覧
| # | 深刻度 | 分類(A〜F) | 指摘 | 根拠 | 修正案 | evidence_level |

## 3. 良い点(維持すべきこと)

## 4. ユーザー判定と対応結果

## 5. チェックリスト更新の提案
(今回の指摘のうち review-checklist.md のFに追記すべき頻出パターン)
```

## 品質基準

- 指摘ゼロのレビューを安易に出さない。指摘がない場合は「何をどう確認して問題なしと
  判断したか」を明記する。
- 一般論の指摘(「異常系を増やすべき」)で終わらせない。どのケースIDの隣に何が
  欠けているか、まで特定する。
