# 共通規約(全スキル必読)

このファイルは `.agents/skills/` 配下の全QAスキルが従う共通ルール。
各スキルは実行開始時にこのファイルを読み込むこと。

## 1. 言語と文体

- 出力・質問・成果物はすべて日本語。
- 成果物はMarkdown。構造化データ(分類結果・カタログ等)はYAMLブロックを併用する。

## 2. ユーザーとの対話ルール

ユーザーへの質問は、自由記述を求めず**選択肢から選ぶだけで完結する形**にする。
属人化を避け、誰が使っても一定水準の結果になるようにするため。

- 選択式質問ツール(`askQuestions`)が利用できる環境では、それを使う。
- 別名の同等ツール(例: `AskUserQuestion`)しかない環境では、それを使う。
- 質問ツールが一切ない環境では、番号付き選択肢をテキストで提示し、番号での回答を待つ。

質問の設計ルール:

- 1回の質問につき選択肢は2〜4個。常に「その他(自由記述)」相当の逃げ道を用意する。
- 複数選択が自然な質問(例: インプット資料の種類)は複数選択可にする。
- 推奨がある場合は先頭に置き「(推奨)」を付ける。

## 3. セッション管理(qa-session.json)

- 作業状態は作業ディレクトリ直下の `qa-output/<セッション名>/qa-session.json` に保存する。
- スキーマは [session-schema.md](session-schema.md) を参照。
- 各フェーズの開始時・完了時・承認時に必ず更新する。
- 中断からの再開時は、まず qa-session.json を読み、`plan` の未完了フェーズから続行する。

## 4. 承認ゲート

各フェーズの完了時に、成果物の要約を提示した上で必ずユーザーの承認を取る。
選択肢は次の4つ:

1. **承認して次へ(推奨)**
2. **修正を指示する** — 指示を受けて成果物を修正し、再度承認を求める
3. **このフェーズをやり直す**
4. **ここで中断する** — セッションを保存して終了(再開方法を案内する)

承認されるまで次のフェーズへ進んではならない。成果物は「AIが作ったもの」ではなく
「人間が承認したもの」として次フェーズに渡る。

## 5. 証拠レベル(推測と事実の分離)

分析・レビュー系の出力では、各指摘・各結論に証拠レベルを必ず付ける。

```yaml
evidence_level: confirmed   # 資料・コード・チケットで確認済み
evidence_level: likely      # 状況証拠あり。要確認
evidence_level: hypothesis  # 推測。裏取りが必要
sources:                    # 根拠(チケットID、ファイルパス、仕様書の節など)
  - BUG-1234
  - spec.md#3.2
```

根拠を示せない指摘は hypothesis とし、confirmed と混ぜて並べない。
「もっともらしいが間違っている分析」が最も有害であることを常に意識する。

## 6. 成果物の出力先と命名

```
qa-output/<セッション名>/
  qa-session.json
  01-defect-analysis.md
  02-test-analysis.md
  03-criteria-analysis.md
  04-spec-review-mode1.md
  05-test-planning.md
  06-feature-investigation.md
  07-code-review.md
  08-test-viewpoint.md
  09-spec-review-mode2.md
  10-test-case.md
  11-test-data.md
  12-test-design-review.md
  99-improvement.md
```

- 番号はフェーズ実行順。スキップされたフェーズは欠番でよい。
- 前フェーズの成果物は、次フェーズ開始時に必ず読み込む(バケツリレー)。

## 7. 不明点の扱い

- インプット資料に無い情報を勝手に補完しない。「不足情報」として成果物に明記する。
- 仕様の曖昧さを発見したら、その場で解釈を決めず `qa-spec-review` の検出フォーマットで記録する。
