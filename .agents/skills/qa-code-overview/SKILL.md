# コード概要作成(QA向け把握資料)

ソースコードから「開発仕様書」を復元するのではなく、**テスト分析前のQAエンジニアが
対象を把握するための読み物**を作るスキル。読者はコードを読まない前提で、業務の言葉で
書き、すべての記述に出典を付けて後から検証できるようにする。

「何をテストすべきか考え始められる状態」に読者を連れて行くことがゴール。
網羅は目的ではない — 概要の価値は取捨選択にある。詳細が必要な領域は深掘りスキルへの
導線として示せばよい。

## 実行前に読むこと

- [_shared/conventions.md](../_shared/conventions.md)(対話ルール・承認ゲート・証拠レベル)
- [_shared/references/domain-glossary.md](../_shared/references/domain-glossary.md)
- (外部KB連携時のみ)[_shared/knowledge-base.md](../_shared/knowledge-base.md)

## 隣接スキルとの違い(脱線防止)

| スキル | 目的 | 粒度 |
|---|---|---|
| **本スキル** | テスト分析前の把握資料 | 概要。読んで理解する |
| qa-feature-investigation | テスト観点の材料となる事実の抽出 | 詳細。バリデーション・権限を網羅 |
| qa-code-review | 品質リスクの検出(シフトレフト) | 指摘単位 |

バリデーション仕様の網羅が必要になったら本スキルで頑張らず qa-feature-investigation へ、
品質リスクの指摘をしたくなったら qa-code-review へ回す。本スキルの成果物は両スキルと
qa-test-analysis のインプットになる。

## 入力

- 対象コード(リポジトリ全体 / PR・diff / 特定機能・モジュール)
- (あれば)設計書・仕様書・マニュアル・PR説明・関連チケット・DBスキーマ

## 手順

### 1. スコープ確認(選択式)

1. **把握したい対象**(単一選択。PR・diffが提示されていればモードBを推奨にする)
   - 今回の改修・PRの内容 → **モードB(改修概要)**
   - 製品・コードベース全体 → **モードA(プロダクト概要)**
   - 特定の機能・モジュール → **モードC(機能概要)**
2. **コード以外に併読する資料**(複数選択): 設計書 / 仕様書 / マニュアル /
   PR説明・チケット / DBスキーマ / なし。選択後、各資料のパス・URLを確認する。

### 2. ドキュメント併読(資料がある場合)

コードより先にドキュメントを読み、対象の「地図」を作る。そのうえでコードを読み、
地図を検証する。この順序が重要 — 地図なしでコードから読み始めると、実装の細部に
引きずられて概要にならない。

- ドキュメントの記述とコードの対応(この章はこのモジュール)をメモしながら読む。
- 食い違いを見つけたら、その場でどちらが正しいか判断しない。「要確認事項」として
  記録する(conventions.md §7)。マニュアルが古いのか実装が変わったのかは読者が
  確認すべきことであり、概要作成者が解釈を決めてはならない。

### 3. コード踏査(モード別)

全部を読もうとしない。読まなかった領域は成果物の「未調査領域」に必ず残す
(推測で埋めるくらいなら空白の方が価値がある)。

- **モードA**: README・ビルド設定・ディレクトリ構成 → エントリポイント →
  データの持ち方 → 代表的な業務フローを2〜3本だけ入口から出口まで追う。
  エントリポイントとデータは対象の形態に合わせて読み替える(Webアプリなら
  ルーティング・画面一覧とDBスキーマ、CLIならコマンド定義と入出力ファイル形式、
  バッチならジョブ定義と連携ファイル)。フローは「ユーザーから見た1操作
  (または1コマンド・1ジョブの起動)」を1本と数える。
- **モードB**: PR説明・コミットメッセージ・チケットで「意図」を掴む → diff を
  ファイル単位でなく**機能単位**にグルーピング → 変更されたシンボルの呼び出し元を
  辿って間接影響を特定 → DBマイグレーション・設定値・フィーチャーフラグの変更を確認。
  意図に書かれていない変更、実装されていない意図は要注意事項として拾う。
- **モードC**: 機能の入口(画面・API・バッチ)→ 正常系の流れ → 主要な分岐と状態。
  入力項目単位の詳細には降りない。

### 4. 成果物の作成

モード別フォーマット(下記)で書く。書き終えたら読者の目で読み直す:

- コードを読まないQAエンジニアに通じる言葉になっているか。コード内の用語
  (クラス名・テーブル名)を使う場合は業務上の意味を添える。
- すべての記述に出典(ファイルパス、ドキュメント名+節)があるか。
- 評価・推測を含む記述(食い違い・要確認事項、回帰リスク、勘所)には
  evidence_level を付けたか。事実の対応関係の記述は出典のみでよい。
  勘所は confirmed に絞らなくてよい — 状況証拠なら likely、経験則からの
  推測なら hypothesis と明示して書く方が、確度の高い項目だけ並べるより
  読者の役に立つ。

### 5. 承認ゲート

conventions.md §4 に従い、要約を提示して承認を得る。

## 出力フォーマット(00-code-overview.md)

出力先は `qa-output/<セッション名>/00-code-overview.md`。qa-orchestrator のフェーズ外の
事前把握資料なので、セッション未開始でも単独実行してよい(その場合も同じパス規約で出力する)。

### モードA: プロダクト概要

```markdown
# プロダクト概要: <対象>
## 1. これは何のシステムか(3行要約)
## 2. 主要機能マップ(機能 → 担当モジュール → 出典)
## 3. 代表的な業務フロー(入口 → 処理 → 出力)
## 4. データの全体像(主要エンティティと関係)
## 5. 外部連携・依存
## 6. 設定・権限・環境による挙動差
## 7. ドキュメントとの対応と食い違い(要確認事項)
## 8. QAが押さえるべき勘所(複雑さ・リスクが集中していそうな領域とその理由)
## 9. 用語対応表(コード内用語 → 業務用語)
## 10. 未調査領域と読み方ガイド(深掘りするときの入口)
```

### モードB: 改修概要

```markdown
# 改修概要: <PR・チケット>
## 1. 何のための変更か(意図の要約。出典: PR説明・チケット)
## 2. 変更内容の要約(機能単位)
## 3. 意図と実装の対応(意図にない変更・実装されていない意図があれば明記)
## 4. 影響範囲(直接変更 / 呼び出し元・共有部品経由の間接影響 / DB・設定・外部連携)
## 5. 挙動が変わる操作・画面・帳票(ユーザー視点)
## 6. 回帰リスクが高そうな箇所とその理由
## 7. ドキュメントとの食い違い・要確認事項
## 8. テスト分析へのインプット要約(→ qa-test-analysis)
## 9. 未調査領域
```

### モードC: 機能概要

```markdown
# 機能概要: <機能名>
## 1. 機能の目的と利用シーン
## 2. 画面・API・バッチの構成
## 3. 処理の流れ(正常系の要約)
## 4. 主要な分岐・状態(概要レベル)
## 5. データ・他機能との関係
## 6. ドキュメントとの対応と食い違い(要確認事項)
## 7. QAが押さえるべき勘所
## 8. 未調査領域(入力仕様・バリデーションの詳細は qa-feature-investigation へ)
```

## 品質基準

- 出典のない記述を書かない。コード・資料で確認できたものだけが confirmed。
- 推測で空白を埋めない。「未調査」「要確認」と明記する方が概要として誠実。
- バグの疑い・仕様との齟齬を見つけても、修正提案や品質指摘に脱線しない。
  要確認事項として記録し、qa-spec-review / qa-code-review に回す。
- 分量は読者が30分で読み切れる程度を目安にする。長くなるなら詳細を削って
  「読み方ガイド」(どこを見ればよいか)に置き換える。
