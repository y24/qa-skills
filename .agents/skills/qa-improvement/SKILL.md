# QA改善(スキルセット自己進化)

スキルセット自体を成長させるスキル。セッションの実行結果を、スキルファイルと
参照ナレッジの改善に変換する。**変更は必ず提案 → ユーザー承認 → 適用の順**で行う。

## 実行前に読むこと

- [_shared/conventions.md](../_shared/conventions.md)
- 対象セッションの qa-session.json(`improvement_notes` と `decisions`)
- 全フェーズの成果物

## 改善の対象

| 対象 | 改善の種類 |
|---|---|
| 各スキルの SKILL.md | 手順の抜け、冗長な質問、出力フォーマットの不備 |
| _shared/references/regression-viewpoint-catalog.md | 新しい回帰観点の追記(未反映分) |
| _shared/references/review-checklist.md | 頻出指摘のFセクションへの追記 |
| _shared/references/code-review-viewpoints.md | QAコードレビューで効いた指摘パターンの追記(「自分の頻出指摘」セクション) |
| _shared/references/domain-glossary.md | セッション中に出た用語の追記 |
| _shared/references/defect-taxonomy.md | 分類軸に収まらなかった不具合のラベル追加 |
| _shared/conventions.md | 対話・承認フローの改善 |

## 手順

1. **材料収集**: `improvement_notes`、`decisions`、各フェーズでのユーザーの修正指示
   (何が差し戻されたか)を集める。
2. **ユーザーヒアリング**(選択式): 「今回のフローで一番手間だった箇所」
   「出力の質が低かったフェーズ」「質問が多すぎた/少なすぎたフェーズ」を選んでもらう。
3. **改善案の作成**: 材料ごとに次の形式で提案する。

   ```markdown
   ### 改善案 N: <対象ファイル>
   - **きっかけ**: (セッション中の何が問題だったか。具体例を引用)
   - **変更内容**: (diffイメージ。どの節にどう書き足す/直すか)
   - **期待効果**:
   ```

4. **承認と適用**: 改善案ごとに「適用 / 見送り / 修正して適用」を選択式で確認し、
   承認されたものだけをファイルに反映する。
5. **ナレッジ還元の確認**: 特に回帰観点カタログとレビューチェックリストFセクションへの
   還元漏れがないか最終確認する(ここが蓄積されないとスキルセットが成長しない)。

## 品質基準

- 承認なしにスキルファイルを書き換えない。
- 「もっと丁寧に」のような曖昧な改善を書かない。手順・文言レベルの具体的な変更にする。
- 1セッションの特殊事情を一般ルール化しない(2回以上起きたことを優先する)。
