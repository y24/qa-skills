# テストデータ設計

テストケースを「実行できる」状態にするためのデータ設計スキル。
ケースが要求する条件を漏れなくデータに翻訳する。

## 実行前に読むこと

- [_shared/conventions.md](../_shared/conventions.md)
- 前フェーズの成果物: テストケース一覧(必須)、機能調査(データ構造の出典として)

## 手順

1. **データ要件の抽出**: 全テストケースの前提条件・入力値から、必要なデータ条件を
   一覧化する(エンティティ × 状態 × 量 × 特殊値)。
2. **共有と分離の設計**: 複数ケースで使い回せるベースデータと、ケース専用データ
   (破壊されるデータ・状態が変わるデータ)を分ける。実行順の依存もここで解消する。
3. **バリエーション設計**: 次を明示的にカバーする。
   - 境界値データ(ケースの具体値と一致させる)
   - 0件 / 1件 / 複数件 / 上限・大量データ
   - 異常データ(不正形式・参照切れ・重複・NULL)— 投入方法(正規画面から入らない
     データはSQL等)も書く
   - 日付・期間データ(期首・期末・期跨ぎ・過去年度)は基準日からの相対で定義し、
     実行日に依存しないようにする
4. **作成手段の決定**(選択式): 手作業手順書 / SQL・スクリプト生成 / 既存データの加工。
   スクリプトを書く場合は冪等(何度流しても同じ状態)にする。
5. **個人情報・機密の確認**: 本番データ由来のデータを使う場合はマスキング要否を
   必ずユーザーに確認する。

## 出力フォーマット(NN-test-data.md)

```markdown
# テストデータ設計: <対象>

## 1. データ要件マトリクス
| データ条件 | 対応ケースID | 種別(共有/専用) |

## 2. データ定義
(エンティティごとの具体値。基準日相対で記述)

## 3. 作成手順 / 生成スクリプト

## 4. 投入・リセットの手順(再実行時の戻し方)

## 5. 注意事項(マスキング・環境制約)
```

## 品質基準

- テストケースの入力値とデータ定義の値を一致させる(微妙に違う値を作らない)。
- 「適当なデータを用意する」を書かない。すべて具体値または生成ルールで書く。
- リセット手順のないデータ設計を完成にしない。
