# 探索的テストヒューリスティクス(オラクル・カバレッジモデル・チャーター)

qa-test-viewpoint(観点軸の網羅チェック)と qa-spec-review モード2
(期待結果の根拠づけ)が参照する。
出典: James Bach / Michael Bolton らのコンテキスト駆動テストの定番ヒューリスティクス
(satisfice.com / developsense.com)。

## 1. FEW HICCUPPS — テストオラクル(「これはバグだ」と主張する根拠の言語)

オラクルとは「問題があると気づく手がかり」のこと。期待結果が仕様書に明記されて
いないとき、**何との不整合を根拠にバグと言うのか**を明示するための語彙として使う。

| 記号 | オラクル | 「〜と一致しない」の中身 |
|---|---|---|
| F | 既知の不具合パターン(Familiar) | 過去に見た故障パターンに**似ている**(これだけは「一致したら」疑う) |
| E | 説明可能性(Explainability) | 挙動の理由を自分が説明できない。説明できない挙動は追う価値がある |
| W | 現実世界(World) | 物理法則・地理・日付の論理・ドメインの事実 |
| H | 履歴(History) | 旧バージョンの挙動。意図・周知のない変化は回帰を疑う |
| I | イメージ(Image) | 組織のブランド・プロフェッショナルとしての体裁(誤字、崩れたUI) |
| C | 主張(Claims) | ドキュメント・ヘルプ・ラベル・ツールチップ・営業資料が謳っていること |
| C | 同種製品(Comparable Products) | 競合・類似システム・社内の類似機能の挙動 |
| U | ユーザー期待(User Expectations) | 明文化されていなくても、想定ユーザーが合理的に期待すること |
| P | 製品内整合(Product) | 同じ操作は画面・文脈が変わっても同じ結果になるべき(製品自身との一貫性) |
| P | 目的(Purpose) | 仕様に形式的に合致していても、機能の存在目的を果たしていない |
| S | 標準(Standards) | 業界標準・アクセシビリティガイドライン・API契約・社内規約 |
| S | 法令(Statutes) | 適用される法規制(個人情報保護、会計基準、業法など) |

### 使い方

- **qa-spec-review モード2**: 期待結果が仕様書に辿れない観点に対し、
  「どのオラクルに基づく期待か」をラベル付けして暫定解釈を作る
  (例: `オラクル: History — 旧画面では四捨五入だった`)。
  オラクル由来の期待結果は evidence_level: hypothesis のままにし、
  仕様確認(PO・開発者への確認)の代替にしない。
- **不具合報告時**: 「なんとなくおかしい」を「Claims と不整合(ヘルプには〜とある)」の
  形に言い換えると、指摘の根拠(sources)が明確になる。
- オラクルはすべて可謬(fallible)。複数のオラクルが矛盾することもある
  (例: History と Claims が食い違う)。その矛盾自体を検出項目として挙げる。

## 2. SFDIPOT — プロダクトカバレッジモデル(観点抽出の軸)

「機能一覧だけ見て観点を出すと、機能以外の6軸を丸ごと見落とす」ことを防ぐ
チェック用の7軸。観点を出す軸としても、出し終わった一覧の漏れ確認にも使う。

| 記号 | 軸 | 見るもの |
|---|---|---|
| S | 構造(Structure) | ファイル、DB、設定、API、モジュール、UI要素、サードパーティ製品 |
| F | 機能(Function) | 機能・操作・トランザクション。ハッピーパスと各機能の際(きわ) |
| D | データ(Data) | 入力・出力・保存・変換。型、範囲、形式、エンコード、破損 |
| I | インターフェース(Interfaces) | 外部との接点: API、DB、ファイルシステム、外部サービス、ブラウザ、OS |
| P | プラットフォーム(Platform) | 動作環境: OS、ブラウザ、解像度、ロケール、コンテナ、インフラ |
| O | 運用(Operations) | 実際の使われ方: 業務フロー、中断、同時利用、管理者操作、バッチ・定期ジョブ |
| T | 時間(Time) | タイムアウト、セッション期限、締め・期跨ぎ、うるう年、DST、競合状態、順序 |

### 使い方

- **qa-test-viewpoint のセルフチェック**: 観点一覧を7軸に振り分け、観点がゼロの軸が
  ないか確認する。ゼロの軸は「対象外の理由」を成果物のセクション3に書く
  (暗黙に落とさない)。
- 非機能特性の網羅は [quality-characteristics.md](quality-characteristics.md) が正。
  SFDIPOT は「品質特性(どんな性質を確認するか)」ではなく
  「プロダクトのどこを見るか」の軸であり、両者は直交する。

## 3. テストチャーター書式(探索的テストセッションの切り出し)

観点一覧のうち、期待結果を事前に固定できない領域・探索が向く領域
(新規UI、複雑な状態、仕様の薄い箇所)は、テストケースに無理に展開せず
チャーターとして切り出せる。

> **「<対象> を、<リソース/データ/ツール> を使って、<発見したいリスク情報> のために探索する」**
> (原型: *Explore [target], with [resources], to discover [information]*)

例:

- 「ユーザー登録フローを、境界近傍のメールアドレス入力を使って、
  入力バリデーションとエラー処理のリスクを発見するために探索する」
- 「決済チェックアウトを、複数通貨とネットワーク切断シナリオを使って、
  信頼性と状態管理のリスクを発見するために探索する」
- 「管理者権限画面を、ロール切替と直接URLアクセスを使って、
  認可バイパスのリスクを発見するために探索する」

- 1チャーターは60〜120分で完了する粒度に保つ(長いと焦点がぼける)。
- チャーターの実行結果からバグを主張するときは §1 のオラクルを根拠に使う。
