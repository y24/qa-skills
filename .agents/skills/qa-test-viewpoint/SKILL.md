# テスト観点抽出

テストケースの一段上の抽象度である「観点」(何を・どういう切り口で確認するか)を
網羅的に抽出するスキル。ここの網羅性がテスト設計全体の質を決める。

## 実行前に読むこと

- [_shared/conventions.md](../_shared/conventions.md)
- [_shared/references/test-design-techniques.md](../_shared/references/test-design-techniques.md)(技法の選択基準)
- [_shared/references/quality-characteristics.md](../_shared/references/quality-characteristics.md)(非機能観点)
- [_shared/references/regression-viewpoint-catalog.md](../_shared/references/regression-viewpoint-catalog.md)(過去不具合由来の観点)
- 前フェーズの成果物: テスト分析、品質基準一覧、仕様レビュー(モード1)、機能調査、QAコードレビュー(あるものすべて)

## 手順

1. **機能の分解**: 対象を「機能 × 入力条件 × 状態 × 出力」の粒度に分解し、観点抽出の
   土台となる一覧を作る。機能調査レポートがある場合はその入力仕様・状態一覧を正とする。
2. **技法ベースの抽出**: 分解した各要素に対し、test-design-techniques.md の選択基準に
   従って技法を適用し観点を出す(適用した技法名を観点に記録する)。
3. **品質特性ベースの抽出**: テスト分析で選定された非機能特性ごとに観点を出す。
   品質基準一覧(qa-criteria-analysis の成果物)がある場合、確認フェーズが UT/IT/ST の
   各基準に対応する観点を必ず作り、導出元に基準ID(QC-xxx)を記録する。
   対応する観点のない基準は「テストで確認されない基準」として明示する。
4. **過去不具合ベースの抽出**: 回帰観点カタログから対象機能・類似機能に該当する
   エントリを取り込む(根拠チケットIDを引き継ぐ)。
5. **コードレビュー結果の反映**(QAコードレビューの成果物がある場合):
   「テスト観点への引き継ぎ」の項目を観点として取り込む(品質特性ラベル・出典を保持)。
   「コードで保証済み(テスト軽減候補)」とされた項目は優先度を下げるか対象外とし、
   その根拠(コードレビューの該当箇所)をセクション3に記録する。
6. **抜け漏れセルフチェック**: 次の頻出漏れパターンを最後に確認する。
   - 無効系・権限なし・0件・上限超え
   - 再実行・二重送信・中断からの再開
   - 画面/帳票/エクスポート間の整合
   - 既存機能への回帰(今回変更していないが影響を受ける機能)
7. **優先度付け**: テスト分析のリスク評価と対応づけて High/Medium/Low を付ける。

## 出力フォーマット(NN-test-viewpoint.md)

```markdown
# テスト観点一覧: <対象>

## 1. 機能分解表

## 2. テスト観点一覧
| ID | 機能/要素 | 観点 | 期待結果の要点 | 導出元(技法/特性/BUG-ID/仕様節) | 優先度 |

## 3. 対象外とした観点とその理由

## 4. 期待結果が定義できなかった観点(→ qa-spec-review モード2 へ)
```

## 品質基準

- すべての観点に**導出元**(技法名・品質特性・過去チケットID・仕様書の節)を付ける。
  導出元を書けない観点は、なぜ必要なのかを説明できるまで削らずに hypothesis と印を付ける。
- 期待結果が曖昧な観点を握りつぶさない。セクション4に集めて次フェーズ(仕様レビュー
  モード2)に渡す。
- 観点とテストケースを混ぜない。「〜を確認する」の粒度を保ち、具体値はここでは出さない。
