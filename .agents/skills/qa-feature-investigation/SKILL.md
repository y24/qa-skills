# 機能調査(コード・デザインからの仕様補完)

仕様書には書かれないがテスト設計に必要な事実(実装上の制約・分岐・既定値)を
コードから抽出するスキル。**テスト観点の材料集め**であり、コードレビューではない。

## 実行前に読むこと

- [_shared/conventions.md](../_shared/conventions.md)
- [_shared/references/domain-glossary.md](../_shared/references/domain-glossary.md)
- (外部KB連携時のみ)[_shared/knowledge-base.md](../_shared/knowledge-base.md)
  (KB の features/ rules/ は手順3の仕様書との突合材料に使える。KB由来の事実は likely 扱い)

## 入力

- 対象コード(リポジトリ・ディレクトリ・PR)
- (あれば)画面設計、仕様書、DBスキーマ

## 調査項目

対象機能について以下を抽出する。**すべてに出典(ファイルパス:行、画面名)を付ける**。

1. **入力仕様**: 各入力項目の型・必須/任意・最大長・許容文字・既定値
2. **バリデーション**: クライアント側/サーバー側それぞれの検証内容とエラーメッセージ。
   両側の不一致は要注意事項として挙げる
3. **権限制御**: 操作・表示の権限チェック箇所。チェックがない操作も記録する
4. **状態・ステータス**: 扱う状態の一覧、遷移を制御する条件
5. **分岐・特殊ケース**: フィーチャーフラグ、設定値依存、日付・ロケール依存の分岐
6. **データ影響**: 書き込むテーブル/ファイル、トランザクション境界、外部API呼び出し
7. **UI構成**(デザインがある場合): 画面遷移、表示条件、仕様書との差分

## 手順

1. 対象コードの入口(画面・API・バッチ)を特定し、処理の流れを追う。
2. 調査項目1〜7を順に埋める。読み切れない箇所は「未調査」と明記する(推測で埋めない)。
3. **仕様書との突合**: 仕様書にない実装上の挙動、実装にない仕様上の記述を差分として列挙する。
   これらは qa-spec-review(モード1)の検出対象にもなる。
4. テスト設計に効く発見(隠れた分岐・不一致・チェック漏れの疑い)をハイライトする。

## 出力フォーマット(NN-feature-investigation.md)

```markdown
# 機能調査: <対象>

## 1. 処理の流れ(入口→出口の要約)
## 2. 入力仕様・バリデーション一覧
| 項目 | 型 | 必須 | 制約 | クライアント検証 | サーバー検証 | 出典 |
## 3. 権限制御
## 4. 状態と遷移条件
## 5. 分岐・設定依存
## 6. データ・外部連携への影響
## 7. 仕様書との差分(要確認事項)
## 8. 未調査領域
```

## 品質基準

- 出典のない記述を書かない。コードで確認できたものだけが confirmed。
- 「バグの疑い」を見つけても修正提案に脱線しない。差分・疑いとして記録し、テスト観点に回す。
