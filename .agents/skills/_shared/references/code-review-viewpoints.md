# QAコードレビュー・チェックポイント(品質特性→コードの見どころ)

qa-code-review が特性別走査に使う。[quality-characteristics.md](quality-characteristics.md)
の各特性を「コード上では何を見れば検出できるか」に翻訳したもの。
**このファイルは育てる資産**。実際のレビューで効いた指摘パターンを
qa-improvement 経由で追記していくこと。

## ISO/IEC 25010 主特性

| 特性 | コード上のチェックポイント(例) |
|---|---|
| 性能効率性 | N+1クエリ、ループ内のI/O・API呼び出し、全件取得後のメモリ内フィルタ、インデックスの効かない検索条件(関数適用・前方一致でないLIKE)、大量データ時のページング欠如、不要な再計算 |
| 互換性 | APIレスポンス・ファイル形式の後方互換を壊す変更、DBスキーマ変更にマイグレーション・移行データ対応があるか、共有データを他機能と矛盾する形で更新していないか |
| 使用性 | エラーメッセージが原因と対処を示すか(例外メッセージの素通し)、入力途中データの保護、破壊的操作の確認、クライアント検証とサーバー検証の不一致 |
| 信頼性 | 例外の握りつぶし(catchして無視)、外部呼び出しのタイムアウト・リトライ有無、部分失敗時の後始末(ロールバック・補償処理)、リソースリーク(close/dispose漏れ) |
| セキュリティ | 認可チェック漏れ(ID直指定・URL直叩きで他者データに届くか)、SQL組み立ての文字列連結、出力エスケープ漏れ、機密情報のログ出力・平文保存 |
| 保守性 | テスト容易性を損なう構造のみ見る: 現在日時・乱数・外部I/Oの直接参照(注入不可)、巨大な条件分岐の塊。命名・スタイルには踏み込まない |
| 移植性 | パス・ロケール・タイムゾーン・改行コードのハードコード、環境依存の前提(OS・文字コード) |

## 業務システム向け拡張観点

| 観点 | コード上のチェックポイント(例) |
|---|---|
| 計算正確性 | 金額・数量への浮動小数点型(float/double)使用、丸め処理の方式・タイミングが箇所ごとに不統一、桁あふれ対策、乗除算の順序による誤差、通貨・単位換算の係数管理 |
| データ整合性 | 複数テーブル更新が同一トランザクションか、二重登録防止がアプリ側チェックのみ(DB一意制約なし)か、集計値と明細を別ロジックで計算していないか、更新の競合制御(楽観ロック等) |
| 監査証跡 | 更新系操作に操作者・日時・変更内容の記録があるか、物理削除していないか(論理削除・履歴の要否)、ログに改ざん・欠落の余地がないか |
| 再実行性 | 同じ処理の再実行で二重計上・二重送信にならないか(冪等キー・処理済みフラグ)、途中失敗からの再開手段、リトライ時の副作用 |
| 期間境界 | 日付比較の < と <= の使い分け(境界日がどちらに入るか)、タイムゾーン・時刻切り捨ての扱い、年度・締め日のハードコード、遡及修正データの集計への反映経路 |
| 権限分離 | 承認フローをバイパスできる経路(API直叩き・一括処理)、自己承認の可否、権限チェックがクライアント側のみになっていないか |
| 帳票一致性 | 画面・帳票・エクスポートで同じ数値を別ロジックで計算していないか(計算の二重実装)、書式定義の共通化 |
| 外部出力再現性 | Excel/CSV出力の文字コード・BOM・ロケール依存書式(日付・小数点)の明示指定、数式セル・書式の扱い、同一入力での出力一致 |

## 自分の頻出指摘(ここを育てる)

<!-- 実際のレビューで繰り返し効いた指摘パターンを、特性ラベル付きで追記する。例: -->
- (例)[再実行性] バッチの処理済み判定がステータス列のみで、異常終了時に中間状態のまま残る
- (例)[計算正確性] 明細ごとの丸めと合計後の丸めが混在し、帳票合計と画面合計がずれる

## 使い方

1. qa-code-review の手順2で選定した特性の行だけを走査する(全行を毎回見ない)。
2. チェックポイントに該当した箇所は出典(ファイルパス:行)付きで指摘にする。
3. 該当しないことを確認できた場合も「確認済み・問題なし」として成果物に残せる
   (=コードで保証済み、テスト軽減候補の材料になる)。
