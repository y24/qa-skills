# テスト設計技法リファレンス

qa-test-viewpoint / qa-test-case-design が参照する技法カタログ。
技法ごとにスキルを分けず、入力の型に応じてここから選択する。

## 技法の選択基準

| 入力の型 | 使う技法 |
|---|---|
| 数値・日付・文字数などの範囲 | 同値分割 + 境界値分析 |
| 複数条件の組み合わせで挙動が決まる | デシジョンテーブル |
| 状態・ステータスを持つ機能 | 状態遷移テスト |
| パラメータの組み合わせが爆発する | ペアワイズ |
| ユーザー操作の一連の流れ | ユースケーステスト |
| 過去に似た不具合がある | エラー推測(回帰観点カタログ参照) |

## 1. 同値分割(Equivalence Partitioning)

入力を「同じ振る舞いになるはずのグループ」に分け、各グループから代表値を1つ選ぶ。

- 有効同値クラスと無効同値クラスの両方を必ず挙げる。
- 無効クラスの例: 型違い、空、NULL、範囲外、権限なし、存在しないID。

## 2. 境界値分析(Boundary Value Analysis)

各同値クラスの境界とその両隣をテストする。基本は「境界値、境界値-1、境界値+1」。

- 数値範囲: min-1 / min / max / max+1
- 文字数: 0 / 1 / 最大 / 最大+1
- 日付・期間: 期首 / 期末 / 期跨ぎ / 月末(28,29,30,31日) / 年末年始 / うるう年
- 件数: 0件 / 1件 / 複数件 / ページング境界 / 上限件数

## 3. デシジョンテーブル

条件の組み合わせごとに期待動作を表にする。

```markdown
| 条件           | R1 | R2 | R3 | R4 |
|----------------|----|----|----|----|
| 権限あり       | Y  | Y  | N  | N  |
| データ存在     | Y  | N  | Y  | N  |
| **期待結果**   | 表示 | 空表示 | エラー | エラー |
```

- 条件が4つを超えたら、独立な条件群に分割するかペアワイズに切り替える。
- 「ありえない組み合わせ」も明示的に排除理由を書く(暗黙に消さない)。

## 4. 状態遷移テスト

状態遷移図・遷移表を書き、(1)全遷移を1回ずつ通す、(2)**無効遷移(禁止操作)**を試す。

- 無効遷移こそ不具合の巣。「その状態でそのボタンは押せないはず」を疑う。
- 再実行・二重送信・戻るボタン・タイムアウト後の操作を遷移として含める。

## 5. ペアワイズ(組み合わせテスト)

全組み合わせが多すぎる場合、任意の2因子間の組み合わせを網羅する最小セットに削減。

- まず因子と水準を表で整理してから削減する。
- 過去不具合で「3因子の組み合わせで発生」した実績があれば、その組は3-wayで残す。

## 6. ユースケーステスト

基本フロー、代替フロー、例外フローをそれぞれ1本以上のシナリオにする。

- 「中断して後で再開」「途中でデータが他者に変更される」も代替フローとして扱う。

## 7. エラー推測

経験ベースで「ここが壊れていそう」を挙げる技法。根拠なしで使うと単なる山勘になるため、
必ず [regression-viewpoint-catalog.md](regression-viewpoint-catalog.md)(過去不具合由来の観点)と
組み合わせ、`sources` に過去チケットIDを付けて使う。
