# 仕様レビュー(曖昧性検出)

テスト設計の前後で仕様の曖昧さを潰すスキル。曖昧さを**解釈で埋めるのではなく、
検出して人間に判断を返す**ことが役割。

## モード

| モード | タイミング | 対象 | 主眼 |
|---|---|---|---|
| **1** | テスト設計の前 | 仕様書・要件チケット | 仕様そのものの曖昧さ・未定義 |
| **2** | テスト観点作成の後 | テスト観点・期待結果 | 合否判定できない期待結果、仕様との不整合 |

オーケストレーター経由ならモードは実行計画で指定される。単独起動時は
「仕様書のレビュー(モード1) / テスト観点・期待結果のレビュー(モード2)」を選択式で確認する。

## 実行前に読むこと

- [_shared/conventions.md](../_shared/conventions.md)
- [_shared/references/spec-ambiguity-checklist.md](../_shared/references/spec-ambiguity-checklist.md)(検出カテゴリと記録書式)
- [_shared/references/domain-glossary.md](../_shared/references/domain-glossary.md)(用語の正)
- (モード2のみ)[_shared/references/exploratory-testing-heuristics.md](../_shared/references/exploratory-testing-heuristics.md)(FEW HICCUPPS: 期待結果の根拠づけ)
- (外部KB連携時のみ)[_shared/knowledge-base.md](../_shared/knowledge-base.md)(用語検証を KB の glossary/ rules/ で補強)

## 手順

1. 対象文書を通読し、チェックリストの10カテゴリで走査する。
   - モード1: 全カテゴリを使う。
   - モード2: 特にカテゴリ10(検証不能な期待結果)、2(条件の裏側)、9(既存機能との整合)を重視。
     加えて「観点が仕様のどの記述に対応するか辿れない」ものを検出する。
2. 検出項目を AMB-XXX 形式で記録する(書式はチェックリスト参照)。
   各項目に深刻度(High/Medium/Low)と evidence_level を付ける。
3. High の項目について、可能なら**暫定解釈**(既存機能や慣例からの推測)を付けるが、
   必ず hypothesis と明示し、確認先(PO・開発者など)を提案する。
   モード2で期待結果が仕様書に辿れない場合は、FEW HICCUPPS のどのオラクルに基づく
   期待かをラベル付けして暫定解釈を作る(例: `オラクル: History — 旧画面では
   四捨五入だった`)。オラクル由来の期待結果は hypothesis のままとし、
   仕様確認の代替にしない。複数のオラクルが矛盾する場合はその矛盾自体を AMB として挙げる。
4. 未知の用語・表記ゆれを発見したら domain-glossary.md への追記候補として挙げる。
   外部KB連携時は、文書中の用語が KB の定義・表記と一致するかの検証にも使う
   (カテゴリ1の検出精度が上がる)。KB と文書が矛盾する場合は AMB として検出し、
   KB 未収載の用語は domain-glossary.md の「KB ingest 待ち」節への記録候補とする。
5. 検出結果を深刻度順に提示し、各 High 項目について「確認に回す / 暫定解釈で進める /
   問題なし(誤検出)」を選択式でユーザーに判定してもらう。判定は qa-session.json の
   decisions に記録する。

## 出力フォーマット(NN-spec-review-modeN.md)

```markdown
# 仕様レビュー(モードN): <対象>

## 1. サマリー
- 検出件数: High X / Medium Y / Low Z
- 最重要の3件(1行ずつ)

## 2. 検出一覧
(AMB-XXX 書式。深刻度順)

## 3. ユーザー判定結果
| ID | 判定 | 備考 |

## 4. 用語集への追記候補
```

## 品質基準

- 曖昧さを勝手に解釈してテスト設計に流さない。判断は常に人間に返す。
- 誤検出を恐れず挙げるが、深刻度のインフレをしない(全部Highにしない)。
